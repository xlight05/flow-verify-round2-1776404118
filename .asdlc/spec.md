# Overview

This project is a simple REST API for managing a todo list. It provides a minimal set of endpoints that allow clients to list existing todo items, create new ones, and delete items they no longer need. The goal is to offer a lightweight, predictable, and easy-to-integrate backend service that can be consumed by web, mobile, or command-line clients.

The target users are developers building todo-style applications or learning how to integrate with REST APIs, as well as end users interacting indirectly through client applications. The system prioritizes clarity, consistency, and correctness of the core create/list/delete workflow rather than advanced task-management features.

The high-level approach is to expose a small, well-documented HTTP interface that manages todo resources using standard REST conventions, returns JSON payloads, and enforces basic validation and error handling.

# Capabilities

## Todo Resource Model
- Each todo item must have a unique identifier generated by the system.
- Each todo must include a non-empty title (short text).
- Each todo may optionally include a longer description.
- Each todo must include a completion status (e.g., done or not done), defaulting to not done.
- Each todo must record a creation timestamp.

## List Todos
- Provide an endpoint to retrieve all todo items.
- Return results as a JSON array, with an empty array when no items exist.
- Each returned item must include the id, title, description, status, and creation timestamp.
- Support a consistent ordering of results (e.g., newest first).
- Return an appropriate success status code when the list is retrieved.

## Create Todo
- Provide an endpoint to create a new todo item from a JSON request body.
- Require the title field; reject requests with missing or empty titles.
- Accept an optional description field.
- Ignore client-supplied values for id, creation timestamp, and completion status (the system sets these).
- Return the newly created todo, including its generated id, in the response body.
- Return an appropriate success status code indicating resource creation.
- Return a clear validation error response for malformed JSON or invalid fields.

## Delete Todo
- Provide an endpoint to delete a todo item by its id.
- Return a success response when the deletion completes.
- Return a not-found error response when the specified id does not exist.
- Ensure deleted items no longer appear in subsequent list responses.

## Error Handling
- Return JSON-formatted error responses with a clear error message.
- Use standard HTTP status codes consistently (e.g., 200/201 for success, 400 for validation errors, 404 for not found, 500 for unexpected errors).
- Never expose internal system details in error messages.

## Data Integrity & Persistence
- Persist todos so that they survive across client requests.
- Ensure ids are unique and never reused after deletion.
- Handle concurrent create and delete operations without data corruption.

## Non-Functional Requirements
- All endpoints must respond within a reasonable time under normal load (e.g., under 500 ms for typical requests).
- The API must accept and return JSON with UTF-8 encoding.
- The API must be documented so that developers can integrate without reading source code.
- The system must log requests and errors to support troubleshooting.
- The API must behave consistently and idempotently where applicable (e.g., deleting an already-deleted item returns a predictable response).
